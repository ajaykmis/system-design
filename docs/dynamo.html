📖 READ Operation Deep Dive

1
Key Hashing & Ring Lookup
MD5 hash determines position on ring → identifies preference list of N nodes
2
Coordinator Selection
First healthy node in preference list becomes coordinator (or load balancer routes to any node)
3
Parallel Read Requests
Coordinator sends read requests to N highest-ranked reachable nodes
4
Quorum Wait (R responses)
Wait for minimum R responses before proceeding (typical R=2 for N=3)
5
Version Reconciliation
Compare vector clocks, return causally unrelated versions to client if conflicts exist

🎯 Read Performance Optimization

Read Repair: Coordinator updates stale replicas opportunistically after returning response to client

  • Coordinator waits briefly for outstanding responses
  • Updates any nodes with stale versions
  • Reduces load on anti-entropy protocol

✍️ WRITE Operation Deep Dive

1
Context & Vector Clock
Client provides context from previous read, coordinator generates new vector clock
2
Local Write
Coordinator writes new version locally with updated vector clock
3
Parallel Replication
Send new version + vector clock to N highest-ranked reachable nodes
4
Quorum Confirmation (W responses)
Write considered successful when W-1 additional nodes respond (typical W=2)

⚖️ The Trade-off: Consistency vs Availability

DynamoDB chooses availability over consistency. In network partitions or failures:

  • Writes are never rejected (if at least one node available)
  • Multiple versions may exist temporarily
  • Application must handle conflict resolution
  • Example: Shopping cart merges all concurrent "add item" operations

🎯 Key Learnings & Design Principles

Eventual Consistency Enables Availability

By relaxing consistency requirements, DynamoDB can remain available during network partitions and node failures. The system prioritizes customer experience over perfect consistency.

Application-Level Conflict Resolution

Rather than rejecting conflicting writes, DynamoDB pushes conflict resolution to the application layer. This enables business logic-aware reconciliation (e.g., merging shopping carts).

Symmetric Peer-to-Peer Architecture

Every node has the same responsibilities. There are no master nodes or coordinators, simplifying operations and eliminating single points of failure.

Tunable CAP Theorem Trade-offs

The (N, R, W) configuration allows each application to tune its consistency, availability, and performance characteristics based on business requirements.

Gossip Protocol for Membership

Nodes discover each other through gossip, creating a self-organizing system that adapts to node additions and failures without central coordination.
